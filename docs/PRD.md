# 赛博教授.skill — 产品需求文档（PRD）

## 产品定位

赛博教授.skill 是一个面向科研训练场景的 meta-skill。

它的目标不是“模仿某位教授说几句像样的话”，而是把一位教授的研究谱系、学术判断、指导习惯和问题拆解方式，打包成一个可复用的科研导师型智能体。

该智能体至少应完成三类任务：
1. **知识继承**：基于教授已有论文与公开材料回答问题；
2. **科研指导**：对学生选题、实验、论文、答辩提供步骤化建议；
3. **行动输出**：在合理边界内给出实验表、代码骨架、排错清单、周计划。

---

## 核心概念

### 三层架构

| 层 | 名称 | 职责 |
|---|---|---|
| Part A | Research Memory | 存储该教授的研究主题、代表性方法、术语体系、评价标准 |
| Part B | Advising Persona | 决定其指导风格、批评语气、写作与实验偏好 |
| Part C | Action Playbook | 把指导转化为操作步骤：选题、综述、实验、写作、代码 |

### 运行逻辑

```text
学生发来问题 / 草稿 / 代码 / 实验计划
  ↓
Advising Persona 判断：该教授会先看什么，怎么追问，怎么下判断
  ↓
Research Memory 补充：结合该教授过往研究方法和术语体系给出方向
  ↓
Action Playbook 输出：形成下一步任务、实验矩阵、写作修改、代码骨架
```

## 关键场景

### 场景 1：赛博导师
- 帮学生收敛问题定义
- 判断题目是否太大、太虚、太旧、太散
- 给出 2–3 条更可执行的研究路线

### 场景 2：赛博审稿人
- 指出逻辑跳跃、实验不足、对比不公平、结论越界
- 按“致命问题 / 中度问题 / 轻微问题”分层输出

### 场景 3：赛博实验设计官
- 设计 baseline / ablation / metrics / sensitivity / visualization
- 给出最小实验闭环与失败后的迭代顺序

### 场景 4：赛博代码教练
- 输出最小可运行代码骨架
- 列出数据处理、训练脚本、日志、配置、随机种子等必要组件
- 明确哪些地方必须人工补全

### 场景 5：赛博答辩官
- 模拟开题答辩、预答辩、组会追问
- 连续追问“为什么这是真问题”“为什么你的创新成立”“为什么实验足够”

## 输入材料规范

### 教授侧材料

```yaml
name:        教授姓名或代号
field:       学科
focus:       研究方向
institution: 单位/实验室
style_goal:  希望蒸馏出的角色（导师/审稿人/代码教练/...）
sources:
  - papers
  - homepage
  - lecture_notes
  - interviews
  - grants
  - profile_summary
```

### 学生侧材料
- 论文摘要、引言、方法、实验段落
- 开题报告 / 研究计划
- 数据说明、实验日志
- 报错信息、伪代码、核心模块代码

---

## 生成文件规范

### `research_memory.md`
包含：
- 研究身份
- 核心问题域
- 代表成果与方法谱系
- 常用评价指标与实验套路
- 术语偏好
- 边界条件：哪些结论不能乱推

### `advising_persona.md`
包含：
- 指导语气
- 判断创新性的标准
- 写作偏好
- 实验偏好
- 容易追问的薄弱点
- 对学生常见错误的容忍度

### `playbook.md`
包含：
- 选题收敛流程
- 文献综述模板
- 实验设计 checklist
- 代码生成与复现 checklist
- 论文修改流程
- 答辩模拟流程

---

## 设计原则

1. **证据优先**：能从教授材料中提取的，绝不靠猜。
2. **行动优先**：输出优先是“下一步怎么做”。
3. **边界清晰**：最新外部论文若未提供，不假装读过。
4. **科研真实感**：允许严格、挑剔、保守，不强行鸡汤。
5. **可持续进化**：允许不断追加新论文、新课题、新纠偏。
